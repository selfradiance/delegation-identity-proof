# v0.2 Server-Mediated Scope Enforcement

**Status:** planning pass only
**Scope:** close the v0.1 client-side scope-enforcement gap in the smallest honest way

## Problem Statement

In v0.1, scope is enforced by the local CLI before it calls AgentGate. That is good enough for a proof of concept, but it is not a trustworthy boundary. A delegated agent can skip the CLI and call the delegated execution path directly, which means it can widen or bypass its delegated authority in practice.

## Why v0.1 Is Not Enough

v0.1 proves bounded delegation logic and economic accountability, but the scope gate is local governance only. The missing piece is a server-side checkpoint that the delegated action must cross before AgentGate execution happens.

## v0.2 Thesis

The local delegation system only recognizes, accounts for, and bounds delegated actions that pass through a narrow checkpoint service in this repo. That service sits directly in front of the delegated execution call, validates delegation state and numeric scope limits, and only then forwards the request to AgentGate. Direct AgentGate calls outside that checkpoint are outside the delegation system's accounting and are not treated as delegated in-scope actions. AgentGate itself stays semantically unchanged.

## Proposed Architecture

```text
Human CLI / local admin paths
        |
        v
  local delegation DB (SQLite)

Delegated agent runtime
        |
        | signed request
        v
Delegation checkpoint service
  - validates delegation and scope
  - records reservation / rejection
  - holds delegated execution path
        |
        v
AgentGate client
        |
        v
AgentGate execute_bonded_action
```

### Where the checkpoint sits

The enforcement checkpoint is the HTTP boundary immediately before `execute_bonded_action`. It is not a general proxy for the whole system. Its job is only to decide whether a delegated action request is allowed to cross from local delegation state into AgentGate execution.

### Credential Boundary

v0.2 uses a local execution credential boundary, not a claim of global network blocking. The checkpoint service holds the local AgentGate execution credential boundary used by this repo for delegation-managed execution. The delegated runtime authenticates to the checkpoint, but it does not receive the local execution path this repo uses to submit delegation-recognized actions.

This is the honest guarantee: the checkpoint defines what this delegation system counts as a delegated action. If a caller has some other valid way to talk to AgentGate directly, that call may still happen, but it is outside this system's delegation accounting and is not treated as a delegated in-scope action.

## Enforcement Responsibilities

For `POST /delegated-actions/execute`, the checkpoint validates, in order:

1. The delegation exists.
2. The caller is the bound delegate for that delegation.
3. The delegation is active for execution:
   `status` is `accepted` or `active`, not revoked, not expired, not terminal.
4. The requested `action_type` is in `allowed_actions`.
5. The requested `declared_exposure_cents` is within `max_exposure_cents`.
6. The projected cumulative effective exposure stays within `max_total_exposure_cents`.
   Effective exposure continues to use the existing `ceil(exposure * 1.2)` rule.
7. The projected action count, including in-flight reservations, does not exceed `max_actions`.

If validation passes, the checkpoint creates a local reservation, forwards the request to AgentGate, and finalizes or reverts the reservation using the existing two-phase pattern. If validation fails, nothing is sent to AgentGate.

Simultaneous delegated requests against the same delegation are serialized by the existing short SQLite write transaction used for reservation before any AgentGate call. That reservation immediately consumes an action slot and effective exposure in `delegation_actions`, so concurrent requests see the updated totals and are rejected if they would exceed `max_actions` or `max_total_exposure_cents`.

## Data And State Implications

This pass should reuse as much v0.1 state as possible.

- `delegations` remains the source of truth for lifecycle status.
- `delegation_actions` continues to track executed actions and in-flight reservations.
- `delegation_events` records checkpoint rejections with explicit reason codes.
- No AgentGate schema or semantics change is required.
- The only new operational requirement is that the checkpoint service, not the delegated runtime, owns the delegation-bound execution path.

This is intentionally narrower than adding a general authorization service or new cross-service policy store.

## High-Level Request / Response Shape

### Request

```json
{
  "delegation_id": "uuid",
  "action_type": "email-rewrite",
  "declared_exposure_cents": 83,
  "payload": {
    "input": "..."
  },
  "auth": {
    "delegate_id": "ed25519-public-key",
    "timestamp": "2026-04-20T15:30:00Z",
    "signature": "base64"
  }
}
```

The request is signed by the delegate identity already bound to the delegation. The checkpoint verifies that signature before it evaluates scope.

### Auth Behavior

In scope for v0.2:

- verify an Ed25519 signature from the bound `delegate_id`
- verify the signature over the delegated execution request fields and timestamp
- reject requests whose timestamps fall outside a short freshness window

Recommended narrow default: accept timestamps only within +/- 300 seconds of server time.

Explicitly not in scope for v0.2:

- nonce storage or replay tracking beyond the timestamp freshness window
- key rotation flows
- multiple active delegate signing keys per delegation
- session/token layers beyond the signed request itself

### Success response

```json
{
  "ok": true,
  "delegation_action_id": "uuid",
  "agentgate_action_id": "uuid",
  "status": "forwarded"
}
```

### Rejection response

```json
{
  "ok": false,
  "rejection_code": "ACTION_TYPE_NOT_ALLOWED",
  "message": "Action type is outside delegated scope."
}
```

## Rejection Codes

The checkpoint should return explicit, stable rejection codes at minimum:

- `DELEGATION_NOT_FOUND`
- `DELEGATE_MISMATCH`
- `SIGNATURE_INVALID`
- `TIMESTAMP_OUT_OF_WINDOW`
- `DELEGATION_NOT_ACTIVE`
- `DELEGATION_REVOKED`
- `DELEGATION_EXPIRED`
- `DELEGATION_TERMINAL`
- `ACTION_TYPE_NOT_ALLOWED`
- `PER_ACTION_EXPOSURE_EXCEEDED`
- `MAX_TOTAL_EXPOSURE_EXCEEDED`
- `MAX_ACTIONS_EXCEEDED`

Downstream AgentGate failures are not scope rejections. They should be surfaced separately as forwarding failures after the checkpoint pass.

## Proof / Demo Plan

The proof path for v0.2 should stay small and outsider-legible:

1. Create and accept a delegation with an allowlisted action and clean numeric limits.
2. Submit one in-scope delegated action through the checkpoint.
3. Observe success: local reservation recorded, AgentGate action created, delegation remains valid.
4. Submit one out-of-scope delegated action through the same checkpoint.
5. Observe rejection with `ACTION_TYPE_NOT_ALLOWED`, no AgentGate action created, rejection logged locally.
6. Submit an action that exceeds `max_exposure_cents` or pushes cumulative effective exposure over `max_total_exposure_cents`, and observe the matching rejection code.
7. Show one direct AgentGate execution outside the checkpoint, or otherwise inspect a direct call path, and show that it produces no checkpoint reservation/event and is not counted against delegation totals.

That is enough to prove the v0.1 bypass gap is materially narrower for the delegation-managed path without claiming a broader authorization system or a global block on all direct AgentGate calls.

## Out Of Scope

This pass does not include:

- recursive delegation chains
- transparency log or tamper-evidence
- human-bond slashing flow
- generalized authorization or policy-engine features
- AgentGate core changes
- multi-service orchestration
- blockchain or public anchoring
- UI work

## Main Risks And Guardrails

### Main risk

The biggest scope-creep risk is turning the checkpoint into a broad authorization platform. That would slow delivery and blur the claim.

### Guardrails

- One checkpointed path only: delegated action execution.
- Keep scope numeric and allowlist-based; do not add resource-pattern policy in this pass.
- Keep SQLite as the local source of truth.
- Keep AgentGate unaware of delegation semantics.
- Reuse the existing two-phase reservation/finalization pattern instead of inventing a new coordinator.

## Recommended Build Sequence

1. Add the checkpoint service with one execution endpoint and one health endpoint.
2. Route delegated action execution through the checkpoint instead of directly to AgentGate.
3. Reuse existing delegation/state validation logic inside the checkpoint.
4. Add explicit rejection-code responses and event logging.
5. Add one success test, one out-of-scope rejection test, and one exposure-limit rejection test.
6. Add one proof note or fixture showing that a direct AgentGate call is distinguishable from a checkpoint-mediated delegated action.
